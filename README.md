# Binary Vulnerability Finder — Руководство по использованию файлов

Добро пожаловать в проект Binary Vulnerability Finder! Это руководство поможет вам понять структуру репозитория, найти модули алгоритмов, тестовые файлы и эффективно вносить свой вклад в разработку.

---

## Обзор структуры репозитория

Основной упор в репозитории сделан на эффективную реализацию поиска уязвимостей в бинарных файлах и комплексную инфраструктуру тестирования:

```text
binary-vuln-finder/
├── efficient/                      # Основная реализация (оптимизированные алгоритмы)
├── integration_tests/              # Набор тестов для гибридного алгоритма
├── elf_files/                      # Тестовые бинарные файлы для проверок
├── cpp files for tests/            # Исходные файлы C++ для компиляции тестов
├── juliet-dynamic/                 # Тест-комплект бенчмарков Juliet
├── naive/                          # Первоначальные реализации 
├── run_experiment.py               # Главный скрипт запуска экспериментов
├── run_angr_juliet.py              # Скрипт запуска тестов Juliet с использованием Angr
└── main.py                         # Точка входа в проект
```

##  Контекст проекта

В данном репозитории реализован оптимизированный подход к поиску уязвимостей в бинарных файлах. Директория `/naive` содержит более ранние  реализации, созданные для исследования базовых алгоритмических подходов. Тем не менее, основным фокусом проекта и рекомендуемой отправной точкой для внесения изменений является папка `/efficient`, содержащая готовые реализации.

## Эффективная реализация алгоритмов

Ядро проекта находится в директории `/efficient` и обеспечивает производительное и масштабируемое обнаружение уязвимостей.

### Модули эффективной реализации (`/efficient`)

Эффективная реализация предлагает оптимизированные алгоритмы для поиска уязвимостей в бинарных файлах. Это рекомендуемый подход для практического использования и главный приоритет проекта:

| Файл | Назначение | Ключевые особенности |
| :--- | :--- | :--- |
| **scanner.py** | Модуль сканера уязвимостей | Быстрое распознавание известных последовательностей инструкций и сигнатур уязвимостей, эффективные алгоритмы поиска |
| **static_analysis.py** | Модуль статического анализа бинарных файлов |Построение графа потока управления, анализ потока данных, обнаружение опасных операций (переполнение буфера, использование после освобождения и т.д.) |
| **dynamic_analysis.py** | Модуль динамического анализа | Обнаружение уязвимостей на основе выполнения и мониторинга поведения программы |
| **bfs_baseline.py** | Базовая линия на основе обхода в ширину (BFS) | Взята из фреймворка Angr — используется для сравнения производительности |

### Как использовать эффективные модули

```python
import angr
from efficient.scanner import VulnerabilityScanner
from efficient.static_analysis import MPCPlanner
from efficient.dynamic_analysis import DirectedSearch

# Пример: использование сканера уязвимостей
project = angr.Project("./binary", auto_load_libs=False)
scanner = VulnerabilityScanner(project)
is_vulnerable = scanner.check_rip_control(state)

# Пример: использование статического анализа (MPCPlanner)
planner = MPCPlanner(project)
results = planner.find_all_paths(main_addr, target_addr)
corridor = results['corridor']  # Коридор достижимости
milestones = results['milestones']  # Контрольные точки
uncertain = results['uncertain']  # Неопределённые блоки

# Пример: использование динамического анализа (DirectedSearch)
# DirectedSearch - это техника исследования для управляемого символического выполнения
search = DirectedSearch(sorted_milestones, final_target)
simgr = project.factory.simgr()
simgr.use_technique(search)
simgr.run()
```

### Сравнение производительности: Эффективная реализация vs Базовый метод BFS

Для сравнения производительности в проект включена базовая реализация BFS (обход в ширину) из фреймворка Angr (`efficient/bfs_baseline.py`). Эта базовая линия добавлена исключительно как отправная точка для сравнения и не разрабатывалась в рамках данного проекта — она служит стандартной метрикой для оценки результатов.

Обе реализации доступны в `/integration_tests/` для валидации и замера производительности.

## Тестовые файлы и тестирование

### Организация тестов

Тесты организованы в директории `/integration_tests`:

| Тестовый файл | Назначение | Что тестирует |
| :--- | :--- | :--- |
| **mpc_dse_test.py** | Управляемый динамический анализ | Символическое выполнение DirectedSearch на основе контрольных точек |
| **static_analysis_test.py** | Интеграционный тест для статического анализа | Алгоритмы статического анализа из эффективного модуля |

### Запуск тестов для эффективной реализации

Чтобы запустить интеграционные тесты, проверяющие работу эффективных алгоритмов:

```bash
# Активируйте виртуальное окружение
source venv/bin/activate

# Запустите тест статического анализа 
python3 integration_tests/static_analysis_test.py

# Запустите тест динамического анализа (MPC + DSE) - занимает 5-10 секунд
python3 integration_tests/mpc_dse_test.py
```

Интеграционные тесты сравнивают работу эффективных реализаций с тестовыми данными и проверяют корректность их выполнения.

### Тестовые данные

* Запускайте из корневой директории проекта
* Виртуальное окружение с установленным Angr (venv/bin/activate)


Ожидаемый результат:

Статический тест: показывает анализ коридора (достижимые блоки) и обнаружение уязвимостей.

Тест MPC+DSE: отображает прогресс навигации контрольных точек и результаты символического выполнения.

Тестовые данные распределены по следующим папкам:
* `/elf_files` — Скомпилированные бинарные файлы ELF для тестирования.
* `/cpp files for tests/` — Исходные файлы C++, которые компилируются в бинарники ELF для тестов.
* `/juliet-dynamic/` — Бенчмарки тест-комплекта Juliet для тестирования динамического анализа.



### Добавление новых тестов

При добавлении новых тестов:

1. Создавайте файлы тестов в `/integration_tests` с паттерном имени `<функция>_test.py`
2. Используйте существующую структуру тестов как шаблон (автономный скрипт Python с функцией run)
3. Включайте ссылки на тестовые данные на файлы ELF в `elf_files/`
4. Добавляйте документацию, объясняющую, что тест валидирует

```python
import angr
import os
import sys

sys.path.append(os.path.abspath(os.path.join(os.path.dirname(__file__), '..')))

from efficient.static_analysis import MPCPlanner

def run_integration_test():
    """Интеграционный тест для вашей новой функции"""
    
    binary_path = "./elf_files/test_mpc_1_x86"
    
    if not os.path.exists(binary_path):
        print(f"[-] Бинарный файл не найден в {binary_path}")
        return
    
    # Инициализируем проект Angr
    project = angr.Project(binary_path, auto_load_libs=False)
    
    # Ваш код теста здесь
    planner = MPCPlanner(project)
    results = planner.find_all_paths(main_addr, target_addr)
    
    print(f"[+] Тест пройден: {results}")

if __name__ == "__main__":
    run_integration_test()
```

Решение проблем с тестами


Проблема: "Binary not found at ./elf_files/test_mpc_1_x86"
* Решение: Убедитесь, что вы запускаете тесты из корневой директории проекта, а не из папки integration_tests


Проблема: "ModuleNotFoundError: No module named 'angr'"
* Решение: Сначала активируйте виртуальное окружение: source venv/bin/activate


Проблема: Тест работает очень медленно
* Решение: Это нормально для тестов DSE - они выполняют символическое выполнение, которое может занять несколько секунд или минут в зависимости от сложности бинарного файла


Проблема: Появляется предупреждение об Unicorn engine
* Решение: Это некритично. Предупреждение "failed loading unicornlib.dylib" - это просто информационное сообщение и не влияет на результаты тестирования

## Подходы к алгоритмам в данном проекте

### Основной: Гибридный алгоритм

Гибридный подход является главным фокусом данного репозитория и находится в папке `/efficient`. В нем реализованы оптимизированные алгоритмы для:

* Статического анализа бинарных файлов с эффективным построением графа потока управления (CFG).
* Динамического поиска уязвимостей на основе выполнения программы.
* Сопоставления с паттернами уязвимых блоков и анализа потока данных.

### Справочный: Наивные реализации
Директория `/naive` содержит первоначальные реализации, исследующие базовые алгоритмические подходы. Они предоставлены исключительно для ознакомления и понимания того, как развивались методы проекта.
